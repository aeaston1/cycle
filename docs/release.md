# Release And Homebrew

Cycle's public install path should be Homebrew:

```sh
brew install aeaston1/tap/cycle
```

The tap owner/name is intentionally fixed. The project repo owner/name should
not be hardcoded into public docs or generated config outside release metadata.

## Release Artifacts

Build a versioned release artifact with:

```sh
mix release.artifact v0.1.0
```

The command writes `dist/cycle-v0.1.0.tar.gz` and
`dist/cycle-v0.1.0.tar.gz.sha256`. The archive contains:

- `bin/cycle`
- `README.md`
- `docs/`
- `packaging/homebrew/cycle.rb` as a draft formula reference
- license file

`bin/cycle` in the archive is a packaged executable, not the source-tree
development wrapper.

The artifact task extracts the archive before writing the checksum and scans it
for known fake secret values and private repository names. Any finding is a
release blocker.

Later releases can add compiled binaries if Cycle gains a compiled daemon.

## Versioning

Use semantic versions:

```text
v0.1.0
v0.2.0
v1.0.0
```

Pre-1.0 releases may change internals, but CLI command names and config paths
should stay stable once documented.

## Homebrew Formula

The production formula belongs in the tap repository:

```text
Formula/cycle.rb
```

The formula should:

- point at a versioned Cycle release artifact
- verify `sha256`
- install `bin/cycle`
- install documentation as package docs
- depend on the Erlang runtime required by the escript package
- depend on required operator runtime tools such as `codex`, `git`, and `mise`
- avoid embedding secrets, local paths, or private repo names

The formula should not install or start the Cycle service automatically. Service
installation belongs behind `cycle service install`.

The checked-in `packaging/homebrew/cycle.rb` is a formula template. Keep
placeholder `OWNER/REPO`, release URL, version, and checksum values until a
matching public release artifact has been published.

## Release Flow

Use this checklist for each public release. Replace `vX.Y.Z` with the release
tag and `X.Y.Z` with the version without the leading `v`.

### 1. Preflight

Confirm the release starts from the intended branch and commit:

```sh
git branch --show-current
git rev-parse --short HEAD
git status --short
```

`git status --short` must print no tracked or untracked release changes. Do not
release from a dirty working tree.

Confirm the Cycle version and Symphony engine lock separately:

```sh
./bin/cycle --version
./bin/cycle symphony path
```

The Cycle package version is the Homebrew and release artifact version. The
Symphony engine lock belongs to Cycle-owned engine registry/config state and
must not be inferred from the Cycle package version. If the release changes the
default engine lock, document that as a separate operator decision in the
release notes.

### 2. Tests

Run the local release gates:

```sh
mix test
tests/smoke.sh
./bin/cycle doctor
```

Run the documented CLI checks:

```sh
./bin/cycle --version
./bin/cycle help
./bin/cycle linear configure --from-env
./bin/cycle symphony path
./bin/cycle project opt-in --repo https://github.com/OWNER/REPO.git
```

Run the manual dry-run checks where prerequisites are available:

```sh
./bin/cycle linear configure --print
./bin/cycle service status
./bin/cycle service install --dry-run
./bin/cycle start --dry-run
```

If a release host intentionally lacks Linear auth or a healthy default engine,
record the refusal message. Do not install services or mutate existing Symphony
services to make validation pass.

When Linear auth is configured, also run:

```sh
./bin/cycle project discover --limit 5
./bin/cycle status
```

Confirm `cycle linear configure --from-env` writes
`${XDG_CONFIG_HOME:-~/.config}/cycle/config.yaml`. Legacy `config.env` is
compatibility-only and should not be generated by new release-path commands.
`project discover` should not depend on non-standard tools that are absent on a
normal Homebrew install.

### 3. Artifact And Checksum

Create the release tag only after preflight and tests pass:

```sh
git tag vX.Y.Z
```

Build the archive:

```sh
mix release.artifact vX.Y.Z
```

Confirm the expected files exist:

```sh
test -f dist/cycle-vX.Y.Z.tar.gz
test -f dist/cycle-vX.Y.Z.tar.gz.sha256
cat dist/cycle-vX.Y.Z.tar.gz.sha256
```

Verify the checksum before publishing:

```sh
shasum -a 256 -c dist/cycle-vX.Y.Z.tar.gz.sha256
```

The artifact task must pass its archive extraction and security scan. Treat any
secret, local state, log, engine checkout, private repository name, or
machine-local config finding as a release blocker.

### 4. Publish

Push the tag and publish the artifact only after the checksum is verified:

```sh
git push origin vX.Y.Z
```

Attach these files to the public release:

```text
dist/cycle-vX.Y.Z.tar.gz
dist/cycle-vX.Y.Z.tar.gz.sha256
```

Do not publish a Homebrew tap update before the release artifact URL and
checksum are available.

### 5. Tap Update

In the Homebrew tap repository, update `Formula/cycle.rb` in a separate commit.
Use the published artifact URL and checksum:

```ruby
url "https://github.com/OWNER/REPO/releases/download/vX.Y.Z/cycle-vX.Y.Z.tar.gz"
sha256 "SHA256_FROM_dist/cycle-vX.Y.Z.tar.gz.sha256"
version "X.Y.Z"
```

Review the local template while preparing the tap commit:

```sh
sed -n '1,220p' packaging/homebrew/cycle.rb
```

Keep the tap commit separate from Cycle repository release-preparation commits
so install changes are easy to audit and rollback.

### 6. Install Test

On a test host or clean Homebrew prefix, install from the tap:

```sh
brew update
brew install aeaston1/tap/cycle
cycle --version
cycle doctor
```

Confirm the installed version matches `X.Y.Z` and `cycle doctor` reports the
expected operator prerequisites without exposing secrets.

### 7. Rollback

If install validation fails after the tap update:

```sh
brew uninstall cycle
```

Revert the tap commit that changed `Formula/cycle.rb`, then reinstall the
previous known-good formula:

```sh
git revert <tap-commit-sha>
brew update
brew install aeaston1/tap/cycle
cycle --version
cycle doctor
```

If the release artifact itself is bad, leave the tag and release notes with a
clear superseded warning instead of replacing the archive in place. Publish a
new patch version with a new artifact and checksum.

If the failure is caused by a Symphony engine regression, roll back the
Cycle-owned engine registry/config lock to the previous Symphony engine version
without changing the Cycle Homebrew package version unless the control plane
artifact is also defective.
